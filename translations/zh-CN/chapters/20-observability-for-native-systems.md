# 原生系统的可观测性

测试与 sanitizer 能减少缺陷逃逸的概率，但无法消除生产故障，也无法解释在真实负载、真实数据、真实依赖失败和真实上线条件下的运行行为。这正是可观测性的用途所在。

在原生系统中，可观测性不足的代价尤为高昂。托管服务停滞时，通常还有运行时提供丰富的异常、堆快照和标准化追踪钩子。C++ 系统可能只剩下符号化不完整的崩溃转储、被某个阻塞依赖卡住的线程池、无法清晰映射到逻辑所有权的 RSS 增长，以及只知道某次部署后延迟翻了三倍的运维人员。如果日志、指标、追踪和转储制品未在事故发生前设计到系统中，调查便只能从猜测开始。

本章讨论的是运行时证据。边界必须清晰。测试回答的是：代码在发布前是否满足契约？sanitizer 和静态分析在开发与 CI 阶段机械地搜索已知缺陷类。可观测性回答的是另一个问题：当原生系统真正运行时，哪些信号能让工程师足够快地解释故障、过载或性能劣化以采取行动？

## 从运维问题出发

可观测性在"加几条日志"的起点上是脆弱的；在"运维问题"的起点上才会变强。

对于一个服务，这些问题可能是：

- 为什么请求延迟升高了，而 CPU 占用却保持平稳？
- 哪些依赖失败正在导致重试、队列增长或部分工作完成？
- 关闭时挂住，是因为工作忽略了取消，还是因为下游依赖从未排空？
- 内存增长是泄漏、碎片化、缓存，还是积压导致的？

对于一个库，问题则会转变：

- 哪个宿主操作触发了失败，输入或版本上下文是什么？
- 库的时间消耗在解析、等待、加锁、分配还是 I/O 上？
- 宿主能否将库的失败与自身的请求或任务标识符相关联？

这些问题决定了哪些字段、指标和 span 值得记录。没有这些问题，团队默认会产出冗长但价值低的遥测：字符串堆砌的日志、没有维度的计数器，或者显示了一切却独独缺少排队、重试和取消的追踪。

### 没有可观测性时调试是什么样子

为了让价值更具体，考虑一个处理文件上传的服务。某用户反映上传超时。以下是在没有结构化可观测性时的调查过程：

```
// 服务的实际日志输出：
[2026-03-14 09:41:02] INFO: Processing upload
[2026-03-14 09:41:02] INFO: Starting validation
[2026-03-14 09:41:32] ERROR: Operation timed out
[2026-03-14 09:41:32] INFO: Processing upload
[2026-03-14 09:41:33] INFO: Starting validation
[2026-03-14 09:41:33] INFO: Validation complete
[2026-03-14 09:41:34] INFO: Upload complete
```

哪次上传超时了？是同一个用户还是另一个用户？在等什么——磁盘 I/O、下游服务、还是锁？有重试吗？重试是对同一文件还是不同文件？这些问题都无法回答。值班工程师只能按时间戳范围 grep 日志，靠猜测建立关联，再请用户重现问题。

现在换用带结构化日志、关联 ID 和维度指标的同一服务：

```cpp
void handle_upload(upload_context& ctx) {
	auto span = ctx.tracer().start_span("handle_upload", {
		{"request_id", ctx.request_id()},
		{"user_id",    ctx.user_id()},
		{"file_size",  ctx.file_size()},
		{"shard",      ctx.shard_id()},
	});

	ctx.log(severity::info, "upload_started", {
		{"request_id", ctx.request_id()},
		{"file_name",  ctx.file_name()},
		{"file_size",  std::to_string(ctx.file_size())},
	});

	auto validation = validate(ctx);
	if (!validation) {
		ctx.log(severity::warning, "validation_failed", {
			{"request_id", ctx.request_id()},
			{"reason",     validation.error().category()},
		});
		ctx.metrics().increment("upload_failures", 1,
			{{"reason", "validation"}, {"shard", ctx.shard_id()}});
		return;
	}

	auto store_result = store(ctx);
	if (!store_result) {
		ctx.log(severity::error, "store_failed", {
			{"request_id",  ctx.request_id()},
			{"dependency",  "blob_store"},
			{"error_class", store_result.error().category()},
			{"latency_ms",  std::to_string(store_result.elapsed_ms())},
		});
		ctx.metrics().increment("upload_failures", 1,
			{{"reason", "store"}, {"shard", ctx.shard_id()}});
		return;
	}

	ctx.metrics().observe_latency("upload_duration_ms", span.elapsed_ms(),
		{{"shard", ctx.shard_id()}});
	ctx.log(severity::info, "upload_complete", {
		{"request_id", ctx.request_id()},
		{"latency_ms", std::to_string(span.elapsed_ms())},
	});
}
```

现在值班工程师按 `request_id` 过滤，看到超时发生在 `store` 阶段且 `dependency=blob_store`，检查按 shard 划分的 `upload_duration_ms` 直方图，发现 shard-3 延迟在 09:40 飙升。blob store 仪表盘确认该依赖在那时出现劣化。调查时间从数小时缩短到数分钟。

差别不在于代码量更多，而在于代码从一开始就带着运维问题编写。

## 日志应当解释决策与状态转换

日志最有价值的场景是记录系统所做的决策以及当时重要的状态，而不是逐函数调用叙述。在原生系统中，这种纪律尤为重要，因为日志的量和开销很快就能成为性能问题。

良好的生产日志是结构化的、稀疏的、稳定的。

- 结构化意味着重要字段以机器可读的键值数据形式输出，而不是埋在叙述性文字中。
- 稀疏意味着默认路径保持安静，异常路径才输出信息。
- 稳定意味着字段名称和含义不会每个迭代周期都漂移。

当操作失败时，日志记录通常应当捕获身份标识、分类信息和当时的局部运行上下文。

- 请求或任务标识符。
- 操作名或路由名。
- 失败类别，而不仅仅是格式化消息。
- 可重试性或永久性（如果代码能判断的话）。
- 影响诊断的资源指标，例如队列深度、shard、对端、或重试次数。
- 当上线状态重要时，还需要版本或构建元数据。

避免两个常见错误。

第一，不要把日志当作指标类问题的唯一信息来源。如果需要了解重试率或队列深度，应将其作为指标发出，而不是强迫运维人员从文本中重建。第二，不要仅因某次事故需要就记录高基数负载或敏感数据块；应将其置于明确的采样或调试路径之后。

`std::source_location` 在低流量的内部诊断或基础设施代码中可以有用，尤其是当需要稳定的调用点标签却不想手工维护字符串时。它不能替代有意义的操作名称。一条写着 `source=foo.cpp:412` 的日志远不如写着 `operation=manifest_reload phase=commit` 的日志有价值。

### 非结构化日志与结构化日志的实际差异

非结构化日志与结构化日志的区别在事故中最为突出——此时读日志的人处于时间压力下，且往往不是写这段代码的人。

```cpp
// 非结构化：人类可读，但对机器不友好。
log("Failed to connect to database server db-prod-3 after 3 retries "
    "(last error: connection refused), request will be dropped");
```

这行包含有用信息，但提取这些信息需要解析英文。如果不使用脆弱的正则表达式，就无法按重试次数、依赖名称或错误类进行过滤。在一个服务集群中，从这类日志聚合失败模式代价高昂且容易出错。

```cpp
// 结构化：相同信息，机器可查询。
ctx.log(severity::error, "dependency_connect_failed", {
    {"dependency",  "db-prod-3"},
    {"attempts",    "3"},
    {"last_error",  "connection_refused"},
    {"action",      "request_dropped"},
    {"request_id",  ctx.request_id()},
});
```

现在 `dependency_connect_failed` 事件可以被计数、按依赖名称过滤，并与特定请求相关联。字段名在代码变更时保持稳定，因此当有人重新措辞某条日志消息时，仪表盘和告警不会随之失效。

## 指标应当追踪吞吐量、饱和度和失败形态

指标能回答日志难以回答的问题：速率、分布、长期漂移，以及跨实例的比较行为。对于原生系统，最有价值的指标通常归属三个族群。

第一类是吞吐量与延迟：请求速率、任务完成率、重试率，以及重要阶段的延迟直方图。延迟应使用直方图，而不仅仅是均值。原生性能故障通常是尾部问题。

第二类是饱和度：队列深度、工作线程利用率、打开的文件描述符、连接池占用率、分配器压力、待处理的定时器，以及未完成的后台任务。这些信号告诉你系统是以健康的方式繁忙，还是在积累无法消化的工作。

第三类是失败形态：按错误类别划分的计数、超时计数、取消计数、解析失败、丢弃的工作、崩溃重启，以及降级模式激活。这些揭示了系统是因依赖缓慢、输入质量变化还是内部背压触发而失败的。

谨慎使用标签。高基数标签是将指标系统变成昂贵负担最快的途径之一。请求 ID、用户 ID、文件路径、任意异常文本和原始对端地址通常不适合作为标签。地域、路由、依赖名称、结果类别和有界的 shard 标识符通常适合。

Gauge 也值得警惕。它们容易添加，也容易误读。如果队列深度 gauge 突然跳升，是短暂突发还是持续趋势？在可能的情况下，将 gauge 与速率或直方图配对，这样运维人员才能判断状态是否正在消退。

## 追踪需要跟随异步所有权，而非仅跟随同步调用

在分布式服务和异步原生系统中，追踪是了解端到端时间去向的唯一实用手段。但 C++ 代码往往因为没能跨执行器、回调、线程跳转和协程挂起保持上下文而损失追踪价值。

如果一个请求进入服务、入队后台工作、等待下游调用，并在之后在另一个 worker 上恢复，追踪应仍能描述一次连贯的操作。这要求在工作所有权跨越时间的边界处显式传播追踪上下文。

这正是前期设计决策的价值所在。结构化并发和显式取消作用域让追踪更干净，因为父子关系本身已有意义。脱离所有者的工作和临时派生的线程会让追踪碎片化为互不相关的 span。

为对应实际等待或服务边界的阶段记录 span。

- 工作开始前在队列中等待的时间。
- 执行本地 CPU 工作的时间。
- 等待下游 I/O 的时间。
- 重试或退避的时间。
- 因取消、关闭或过载丢弃造成的时间损失。

不要为每个辅助函数都创建 span。那只会产生追踪噪声而没有因果价值。追踪的目的是解释延迟结构和依赖形态，而不是复述调用图。

### 没有追踪上下文：隐形的排队时间

异步原生服务中一个常见的失败模式是延迟发生在排队阶段而非执行阶段。如果没有跨执行器边界的追踪传播，这个问题是不可见的：

```cpp
// 没有追踪上下文传播。span 只覆盖执行阶段，不覆盖等待阶段。
void enqueue_work(thread_pool& pool, request req) {
	pool.submit([req = std::move(req)] {
		auto span = tracer::start_span("process_request");  // 工作运行时才开始计时。
		process(req);
	});
	// submit() 到 lambda 实际执行之间的时间丢失了。
	// 如果线程池饱和，请求在队列中等待 500ms，
	// 但追踪显示 2ms 的执行时间。运维人员在追踪中
	// 看到低延迟，而用户体验到高延迟。排队时间是
	// 一个盲点。
}
```

有了正确的上下文传播，完整图景就清晰了：

```cpp
void enqueue_work(thread_pool& pool, request req, trace_context ctx) {
	auto enqueue_time = steady_clock::now();
	pool.submit([req = std::move(req), ctx = std::move(ctx), enqueue_time] {
		auto queue_span = ctx.start_span("queued", {
			{"queue_ms", std::to_string(duration_cast<milliseconds>(
				steady_clock::now() - enqueue_time).count())},
		});
		queue_span.end();

		auto exec_span = ctx.start_span("process_request");
		process(req);
	});
}
```

现在追踪显示两个 span——排队和执行——均与父请求关联。当排队时间占主导时，在追踪瀑布图中立即可见。这类延迟正是仅靠指标（平均处理时间）会系统性隐藏的。

## 崩溃诊断是可观测性的一部分，不是独立的应急工作

原生服务和工具需要在首次崩溃发生前就有崩溃处理方案。这个方案不止于"开启转储"。你需要知道转储存放在哪里、如何符号化、如何映射到精确的构建版本、运维人员如何将其与日志和追踪关联，以及附加了哪些进程元数据。

至少，一个崩溃事件应当能够关联到：

- 精确的二进制或构建 ID。
- 匹配构建生成的符号文件。
- 部署元数据，例如版本、环境和上线环。
- 失败操作前后的最近结构化面包屑。
- 线程身份，以及相关线程的栈回溯（在可能的情况下）。

上一章中的构建工作正是让这一切成为可能的前提。符号服务器、构建 ID 和发布元数据是构建层面的关注点，其运维价值在事故发生于凌晨三点、值班工程师需要有效栈回溯而非原始地址时才显现出来。

崩溃上报还需要策略。某些组件应快速失败，因为继续运行有数据损坏的风险。其他组件可以隔离失败的请求或插件，同时保持宿主进程存活。可观测性应在事后使这一决策清晰可辨。如果进程因不变量违反而主动中止，应在终止前输出足够的上下文，使这次崩溃有别于任意段错误。

## 原生系统中资源可见性更为重要

C++ 服务中反复出现的一个运维问题是将逻辑工作增长与内存缺陷混淆。RSS 上升、延迟增大，所有人都在问是否存在泄漏。有时确实如此，但通常解释并不那么清晰：分配器保留、超大缓存、无界队列、停滞的消费者、mmap 增长、文件描述符泄漏，或新流量模式下的碎片化。

单一指标无法解决这个问题。你需要一组资源信号，将运行时行为与可能的原因关联起来。

- RSS 和虚拟内存，用于粗略了解进程形态。
- 分配器特有统计数据（在可用时）。
- 内存中工作积压的队列深度和积压时龄。
- 打开的文件描述符或句柄计数。
- 活跃线程数和阻塞线程指标。
- 连接池占用率和超时计数。
- 有意保留内存的组件的缓存大小和淘汰指标。

目的不是暴露每个分配器 bin 或内核计数器，而是让可能的失败模式可区分。如果内存上升的同时队列深度和积压时龄也在上升，过载是比纯粹泄漏更强的假设。如果内存上升而队列深度平稳、句柄计数上升，则资源泄漏变得更可信。可观测性应当缩小搜索空间。

## 库需要宿主拥有的遥测边界

可复用库不应假设存在全局日志框架、指标后端或追踪 SDK。这会造成与全书前面讨论的相同的依赖反转问题，只不过现在是以运维形式出现。库应改为暴露一个宿主可实现的窄诊断边界。

### 局部示例：一个面向库的诊断接收器

```cpp
enum class severity { debug, info, warning, error };

struct diagnostic_field {
	std::string_view key;
	std::string_view value;
};

struct diagnostics_sink {
	virtual ~diagnostics_sink() = default;

	virtual void record_event(severity level,
							  std::string_view event_name,
							  std::span<diagnostic_field const> fields) noexcept = 0;

	virtual void increment_counter(std::string_view name,
								   std::int64_t delta,
								   std::span<diagnostic_field const> dimensions) noexcept = 0;
};
```

这类接口让库保持诚实。库可以上报解析失败、重试、缓存淘汰或重新加载时间，而无需将某个厂商 SDK 硬编码到所有使用它的二进制文件中。宿主决定如何附加请求 ID、导出指标，或接入追踪系统。

权衡是刻意的抽象工作。对于只在内部使用的小型组件，直接集成或许可以接受。对于可复用库，宿主拥有的遥测通常是更干净的长期设计。

## 应避免的做法

原生可观测性以几种熟悉的方式走偏。

- 为每次分配、加锁或函数入口都记录日志，寄望于数据越多越安全。
- 将高基数标识符用作指标维度。
- 产出的追踪跟随同步辅助调用，但在执行器或协程边界处丢失异步上下文。
- 发布符号化薄弱的二进制文件，却期望崩溃分析事后能奏效。
- 将日志作为唯一的运维接口，而不是将日志、指标、追踪和转储结合起来。
- 让库遥测依赖特定的服务日志栈。

所有这些都会产生成本，却没有相称的诊断价值。

## 要点

原生系统的可观测性是运行时证据设计。从具体的运维问题出发。用日志记录决策和状态转换，用指标追踪速率和饱和度，用追踪了解端到端延迟结构，用崩溃制品支持事后调试。保留使这些信号有意义的异步和所有权边界。暴露能让运维人员区分泄漏、积压、碎片化和依赖停滞的资源信号。

主要权衡是开销与解释质量之间的取舍。丰富的遥测会增加 CPU、内存、存储和设计复杂性。稀疏的遥测会延长事故持续时间，让原生故障模糊不清。选择能回答真实运维问题的最小信号集，然后让它保持稳定。

复习问题：

- 这个服务或库今天能在不猜测的情况下回答哪些运维问题？
- 哪些日志字段足够稳定和结构化，能够支持自动化，而不仅仅是人工读文本？
- 哪些指标能区分吞吐量、饱和度和失败形态，而不是将它们混在一起？
- 追踪上下文是否能在执行器跳转、回调和协程挂起边界中存活？
- 生产崩溃能否关联到精确的构建身份、符号以及附近的运行上下文？
