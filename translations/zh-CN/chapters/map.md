# 全书导图

本书围绕这样一些位置来组织：当设计含糊时，生产环境中的 C++ 会在这些地方变得昂贵。它从所有权、不变量、失败边界和 API 形状开始，因为后续架构依赖这些选择本身是可评审的。然后它会进入那些改变日常设计的库与语言工具，再进入接口与打包，再进入并发与性能，再进入验证与可观测性，最后落到完整的生产模式。

未定义行为是一条贯穿性的关注点，而不是路线图上的单独一站。所有权章节处理生命周期和别名风险。并发章节处理数据竞争、关闭缺陷以及比其所有者活得更久的工作。数据和性能章节处理失效、布局、局部性，以及薄弱测量带来的虚假信心。验证章节覆盖那些能捕获代码评审单独无法发现问题的工具链和运行时信号。

## 第一部分：核心心智模型

第一部分建立全书其余内容赖以存在的词汇：所有权、生命周期、值语义、不变量、失败边界，以及签名如何传达成本与保留行为。如果这些术语是模糊的，后面的章节就会沦为风格之争，而不是工程之争。

## 第二部分：编写现代 C++ 代码

第二部分覆盖那些应该改变普通 C++23 设计的标准库和语言工具：借用类型、结果类型与替代类型、概念、范围、生成器，以及克制使用的编译期工作。重点不是特性覆盖，而是哪些工具会改变契约、评审负担和成本模型。

## 第三部分：接口、库与架构

第三部分从局部代码进入子系统和包边界。问题变成：一旦引入回调、类型擦除、模块、打包和 ABI 约束，接口是否仍然诚实。这也是局部优雅开始与长期可组合性竞争的地方。

## 第四部分：并发与异步系统

第四部分把并发视为跨时间的所有权。共享状态、协程挂起、取消和背压都被表述为生命周期与吞吐量问题，而不是原语目录。目标是让异步工作有界、有人负责，并且可停止。

## 第五部分：数据、内存与性能

第五部分关注表示决策如何变成运行时行为。数据布局、容器选择、分配策略、局部性和测量纪律被视为一个相互连接的成本故事，而不是彼此分离的优化技巧。

## 第六部分：验证与交付

第六部分覆盖在设计确定之后让原生系统保持诚实的整套栈：针对边界失败的测试、Sanitizer 与静态分析通道、构建诊断，以及运行中系统的可观测性。重点是证据质量，而不是为了检查清单而检查清单。

## 第七部分：生产模式

第七部分把前面的章节应用到完整的工程形态上。第 21 章展示服务边界，在那里所有权、准入控制、关闭和遥测都必须对齐。第 22 章对一个可复用库做同样的事，它的契约必须经得起其他团队的使用。第 23 章以一套评审者工作流收尾，把本书其他部分转化为日常代码评审行为。

## 附录

附录刻意保持紧凑。它们提供面向决策的特性索引、参考工具链与诊断基线、简版评审检查清单，以及锚定本书核心术语的术语表。它们存在是为了加快实际工作，而不是变成第二本教科书。
