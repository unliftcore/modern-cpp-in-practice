# 附录：术语对照表与翻译约定

本附录用于统一简体中文译稿中的术语、语气与表达方式。所有章节都应遵守这里的约定，以保证整套译文在技术含义、措辞风格和读者体验上保持一致。

## 翻译约定

- 面向有经验的程序员，语气直接、克制、技术化，不添加鼓励式口吻或教程式铺垫。
- 优先保留设计压力、约束、权衡、失败模式和工程后果，不把原文改写成松散的概念介绍。
- 代码、类型名、库名、标准术语、文件名、链接路径和 Markdown 结构保持原样；仅翻译读者可见的自然语言内容。
- 对于在中文技术语境中已有稳定译法的术语，采用常见且精确的译名；对容易歧义的术语，首次出现时优先使用本表定义的固定译法。
- 需要强调风险、边界或审查判断时，使用简洁、确定的表述，不弱化原文判断力度。
- 追求“信、达、雅”，但以“信”和“达”为先：先确保语义准确，再追求中文行文自然。

## 核心术语对照

| 英文 | 统一译法 | 说明 |
|---|---|---|
| ownership | 所有权 | 指对资源有效性与释放责任的承担。 |
| borrowing | 借用 | 指不接管生命周期的使用。 |
| lifetime | 生命周期 | 统一译为“生命周期”，避免与“作用域”混用。 |
| contract | 契约 | 指类型或函数在边界上的承诺。 |
| invariant | 不变量 | 保持数学与工程语义一致。 |
| value semantics | 值语义 | 标准译法。 |
| identity | 身份 | 当对象“作为其自身”重要时使用。 |
| failure boundary | 失败边界 | 不译为“故障边界”，以贴近软件错误处理语境。 |
| boundary translation | 边界转换 | 指在边界处转换失败表示。 |
| undefined behavior | 未定义行为 | 缩写仍可写作 UB。 |
| reviewable | 可评审 | 指足够清晰，便于审查者推理。 |
| structured concurrency | 结构化并发 | 采用并发领域常用译法。 |
| cancellation | 取消 | 指主动停止无效工作的请求。 |
| suspension boundary | 挂起边界 | 保留与协程语义的一致性。 |
| contention | 争用 | 用于锁、核心时间、缓存行等竞争。 |
| backpressure | 背压 | 采用业界常用译法。 |
| throughput | 吞吐量 | 标准译法。 |
| locality | 局部性 | 指硬件访问局部性。 |
| cost model | 成本模型 | 指时间、内存、同步等具体成本来源。 |
| hot path | 热路径 | 标准译法。 |
| vocabulary type | 词汇类型 | 指用于表达边界契约的标准类型。 |
| observability | 可观测性 | 标准译法。 |
| diagnostic build | 诊断构建 | 指以发现和解释问题为目标的构建配置。 |
| type erasure | 类型擦除 | 标准译法。 |
| call site | 调用点 | 根据上下文可写作“调用处”，默认使用“调用点”。 |
| shutdown | 关闭 | 指系统或组件进入停止过程。 |
| tradeoff | 权衡 | 统一不用“折中”。 |
| failure mode | 失败模式 | 指设计在什么情况下出错。 |
| ownership transfer | 所有权转移 | 与 move 语义配套使用。 |
| shared state | 共享状态 | 并发章节统一译法。 |
| tooling | 工具链 / 工具 | 结合上下文判断，涉及构建与诊断体系时优先“工具链”。 |
