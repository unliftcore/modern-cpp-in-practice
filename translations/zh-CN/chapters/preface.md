# 前言

这不是一本现代 C++23 语言课程，也不是一份现代化改造日志，而是一份面向生产环境的指南。它写给已经知道软件如何被构建、测试、发布和调试，但需要一种可靠方法在真实设计压力下做出良好 C++ 决策的程序员。主题不是语法覆盖，而是所有权、生命周期、接口形状、失败边界、并发、数据布局、性能证据与验证。

本书假定你不需要别人教你写循环或配置编辑器。你需要的是，如何决定一个函数应该通过 `std::string_view` 还是 `std::span<const std::byte>` 来借用，所有权应该在何处转移，失败应该抛异常还是返回 `std::expected`，什么时候范围管道能澄清算法，什么时候协程帧会变成生命周期风险，以及什么样的证据才足以支撑一项性能结论。如果你想要特性导览、入门脚手架或按头文件分类的目录，这不是那本书。

这种前提是刻意设定的。你不必以 C++ 专家的身份开始阅读，但你必须以工程师的身份开始：已经能够阅读非平凡代码、推理 API、跨子系统追踪控制流，并处理测试、调试、性能工作和工程权衡。本书就是围绕这样的读者组织的。

整部书分为七个部分。第一和第二部分建立让日常代码保持可评审的心智模型与日常词汇。第三部分向外扩展到接口、多态、库、模块和 ABI 现实。第四和第五部分处理并发、数据布局、分配与测量。第六部分讨论让原生系统保持诚实的验证与诊断栈。第七部分以完整的生产形态收尾：一个服务、一个可复用库，以及一套评审者工作流。

之所以采用这样的结构，是因为 C++ 中昂贵的失败很少是局部的。一个服务通过 `shared_ptr` 让请求状态继续存活，把工作分发到后台任务，通过副作用记录失败，并在关闭期间挂起。一个库在边界接受借用视图，然后把它们存储到调用方生命周期之后。一次性能重构去掉了一次拷贝，却在别处悄悄增加了争用和分配器压力。每个局部选择看上去都可能合理。失败发生在所有权、时间、错误传递和成本模型相互作用的时候。

未定义行为是一种相关压力，它贯穿全书，而不是被隔离在某个专门章节中。悬空引用、数据竞争、失效迭代器，或者一个悄悄比其源对象活得更久的视图管道，都可能让一个看似合理的设计只在优化、负载或不同工具链下失败。所有权、并发、性能和工具链章节都会分别讨论与其领域最相关的 UB 风险。应当把对 UB 的警惕视为贯穿全书的线索，而不是一次性的警示标签。

## 如何阅读本书

你可以从头顺序阅读。整个顺序旨在分层建立判断力：先是所有权与不变量，然后是日常使用的库与语言工具，接着是架构、并发、性能、验证，以及完整的生产模式。但如果你已经知道自己在解决哪类问题，这组章节也支持定向切入：

| 起点 | 推荐章节 |
|---|---|
| **设计接口或库边界** | 第 4、9、10、11 和 22 章 |
| **构建或修复原生服务** | 第 1、3、12、13、14、20 和 21 章 |
| **编写泛型且可复用的实现代码** | 第 5、6、7、8 和 22 章 |
| **处理热路径、内存行为或测量** | 第 15、16 和 17 章 |
| **强化验证与构建诊断** | 第 18、19 和 20 章 |
| **评审生产级 C++ 变更** | 第 1、3、4、14、19、22 和 23 章 |

每章都会在靠前位置声明先决条件，因此你可以从问题所在之处进入，只在需要时补充背景。附录是紧凑的辅助材料：面向决策的特性索引、工具链与诊断基线、简版评审检查清单，以及本书核心术语的规范术语表。

主要基线是 C++23。只有在 C++26 会改变你今天会做出的决策，或能消除现有模式中的实质性成本时，本书才会提到它。面向未来的内容存在，是为了让建议对近期现实保持诚实，而不是把本书写成标准演进评论。

如果本书成功，它带给你的不应只是词汇。你应当能够看着一个设计或一份 diff，把权衡说明白：它换来了什么、带来了什么风险、让调用方或运维承担了什么，以及什么证据能证明这个选择是可靠的。这正是“了解现代 C++ 特性”和“把现代 C++ 用好”之间的区别。
