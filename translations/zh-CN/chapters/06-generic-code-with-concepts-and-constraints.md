# 使用概念与约束编写泛型代码

只有当问题中的“家族相似性”是真实存在的时，泛型代码才有价值。如果作者用模板来推迟接口决策，它就会变成破坏性的东西。生产中的问题不是“我该怎么让它可复用？”，而是“我该如何消除重复，同时又不让调用契约、诊断信息和失败行为变得不透明？”

概念是很长一段时间以来，第一个能直接改善这一局面的 C++ 特性，而且改善发生在读者最需要帮助的边界处。它们不会自动让泛型代码变简单。它们让你可以用贴近真实设计的方式，陈述一个模板期待什么。这和“先实例化再祈祷编译器错误能指到正确行”的时代相比，是一次重大转变。

本章聚焦于普通产品团队也能维护的受约束泛型代码：可复用的转换、狭窄的扩展点、策略对象，以及那些假设必须保持可评审的算法家族。

## 从变化开始，而不是从模板开始

大多数糟糕的泛型代码都始于一个错误前提：“这些函数看起来很像，所以我应该把它们模板化。” 表面的语法相似远远不够。真正的问题是：设计中的哪些部分允许变化，哪些不变量必须保持固定。

想象一个内部可观测性库，它会把 metric batch 写到不同的 sink：内存中的测试收集器、本地文件和网络导出器。不变的部分很直接：batch 有 schema、一次 flush 内时间戳必须单调、序列化失败必须报告、关闭时不能丢掉已确认的数据。可变的部分是字节最终写到哪里。

这指向的是一条狭窄的泛型接缝，而不是把整条流水线都模板化的理由。

如果你把从解析到重试逻辑再到传输机制的所有东西都模板化，那你写的就不再是可复用代码，而是在代码库里造第二门语言。只有当变化边界本来就足够诚实时，概念才帮得上忙。

## 约束边界，这样实现才能保持普通

概念在实践中的主要用途，不是花哨的重载排序。它的主要用途，是告诉调用方以及编译器：你的算法可以假设哪些操作存在。

考虑一个 batching 辅助函数，它把已经序列化好的记录写入某个 sink：

```cpp
template <typename Sink>
concept ByteSink = requires(Sink sink,
std::span<const std::byte> bytes) {
{ sink.write(bytes) } -> std::same_as<std::expected<void, WriteError>>;
{ sink.flush() } -> std::same_as<std::expected<void, WriteError>>;
};

template <ByteSink Sink>
auto flush_batch(Sink& sink,
 std::span<const EncodedRecord> batch)
-> std::expected<void, WriteError>
{
for (const auto& record : batch) {
if (auto result = sink.write(record.bytes); !result) {
return std::unexpected(result.error());
}
}
return sink.flush();
}
```

这段代码有几件事做得很好。

- 概念名描述的是角色，而不是实现技巧。
- 被要求的操作很少，而且在运维上有实际意义。
- 失败行为是契约的一部分。
- 函数体是普通代码；实现的抽象程度没有超过问题本身的需要。

另一种写法是经典的无约束模板：

```cpp
template <typename Sink>
auto flush_batch(Sink& sink, const auto& batch) {
for (const auto& record : batch) {
sink.push(record.data(), record.size()); // RISK: hidden, undocumented assumptions
}
sink.commit();
}
```

这个版本看起来更短，但在每一个与生产相关的维度上都更差。它的假设没有说出来。错误契约不清楚。对记录形状的要求是偶然形成的。一旦不匹配，编译器会在使用点制造噪声，而不是清楚地陈述接口是什么。

### 具体看一眼错误信息问题

要理解概念究竟修复了什么，可以看一眼：有人把错误类型传给无约束版本时，会发生什么：

```cpp
struct BadSink {};
BadSink sink;
std::vector<EncodedRecord> batch = /* ... */;
flush_batch(sink, batch);
```

没有概念时，编译器会实例化模板体，并在实现深处失败。主流编译器给出的典型错误大致会像这样：

```
error: 'class BadSink' has no member named 'push'
    in instantiation of 'auto flush_batch(Sink&, const auto&) [with Sink = BadSink; ...]'
    required from here
note: in expansion of 'sink.push(record.data(), record.size())'
error: 'class BadSink' has no member named 'commit'
note: in expansion of 'sink.commit()'
```

这只是一个简单例子，就已经有两个错误了。在生产里，模板很少这么浅。sink 可能会穿过三层 adapter，而每一层又都是模板。真正的错误会出现在一串很深的实例化栈底部，程序员必须在脑中把这条链倒着拆开，才能明白到底哪里错了。再叠上深层嵌套模板和标准库类型后，这种诊断信息动辄就是几十行。

有了 `ByteSink` 概念，同一个错误会在调用点产生一条单一而有针对性的错误：

```
error: constraints not satisfied for 'auto flush_batch(Sink&, ...) [with Sink = BadSink]'
note: because 'BadSink' does not satisfy 'ByteSink'
note: because 'sink.write(bytes)' would be ill-formed
```

错误信息会指出概念名、指出未满足的要求，并把位置定位在调用点，而不是实现内部。程序员会立刻知道 `BadSink` 需要提供什么接口。

### 它替代了什么：SFINAE

在概念出现之前，约束模板的标准技术是 SFINAE（Substitution Failure Is Not An Error）。它的思路是让模板签名本身对错误类型不成立，这样编译器就会在重载决议中静默丢弃它，而不是产生硬错误。

前 C++20 代码里，`ByteSink` 约束的等价写法大致是这样：

```cpp
// SFINAE approach (using enable_if to constrain the same interface)
template <typename Sink,
          std::enable_if_t<
              std::is_same_v<
                  decltype(std::declval<Sink&>().write(
                      std::declval<std::span<const std::byte>>())),
                  std::expected<void, WriteError>
              > &&
              std::is_same_v<
                  decltype(std::declval<Sink&>().flush()),
                  std::expected<void, WriteError>
              >,
              int> = 0>
auto flush_batch(Sink& sink, std::span<const EncodedRecord> batch)
    -> std::expected<void, WriteError>;
```

这是同一组约束，只是换成了没人想读的表达形式。带着 `decltype` 和 `std::declval` 的 `std::enable_if_t` 描述的不是设计意图，而是在利用编译器机制。SFINAE 拒绝这个重载时，典型的错误信息往往只是“没有匹配 `flush_batch` 的函数调用”，却不会说明究竟是哪条要求没满足。当存在多个受 SFINAE 保护的重载时，编译器甚至可能把每一个被拒绝的候选都列出来，却仍然不解释它们各自为什么失败。概念版本在每一个维度上都更好：可读性、错误质量和可维护性。

要强力约束公共表面，这样实现就可以保持无聊。这个权衡是对的。

## 概念应该描述语义，而不只是语法

一个只检查成员名是否存在的概念，确实比没有好，但它仍然可能是个薄弱设计。只有当概念对应的是系统中的某个语义角色时，生产中的泛型代码才会变得可维护。

`SortableRange` 比 `HasBeginEndAndLessThan` 更好。`ByteSink` 比 `HasWriteAndFlush` 更好。`RetryPolicy` 比 `CallableWithErrorAndAttemptCount` 更好。概念越像团队本来就在使用的设计术语，它在代码评审和诊断中就越有用。

这很重要，因为概念服务于两类受众。

1. 编译器用它们来选择和拒绝实例化。
2. 人类用它们来理解算法期待的是哪一类东西。

如果名字和结构只满足了编译器，那它的价值就已经丢掉一半。

这并不意味着概念应该试图证明每一条语义规律。大多数有用的不变量仍然更适合测试，而不是静态强制。`RetryPolicy` 概念可以要求某个调用签名和结果类型，但它无法证明这个策略对某个具体操作是幂等安全的。接受这个边界。把能在接口里检查的内容说清楚，把剩下的部分交给测试和评审验证。

## 优先选择狭窄的定制点，而不是模板蔓延

很多可复用组件并不需要庞大的概念层级。它们只需要一两个经过仔细选择的扩展点。

假设一个存储子系统想支持几种记录类型，它们都可以被序列化到 wire buffer 中。一个常见的糟糕做法，是定义一个主模板，允许特化散落在整个代码库里，再让参数相关查找或隐式转换决定实际发生什么。这会让行为难以发现，也很容易被破坏。

更干净的设计通常是一个狭窄的定制点，并配上显式要求的签名。如果行为归类型本身所有，那可以是成员函数；如果类型应当与序列化库解耦，那可以是非成员操作。无论哪种方式，概念都应该约束形状和结果。

关键在于局部性。评审者应该能够快速回答三个问题。

- 允许变化的到底是什么？
- 哪些不变量保持固定？
- 新的模型类型在哪里 opt in？

如果答案分散在十个头文件里，还要依赖一些偶然形成的重载决议规则，那这个泛型设计就已经过于隐式了。

## 泛型代码不是隐藏成本的许可证

模板以漂亮的调用点掩盖分配、拷贝和代码体积增长，这早就是出了名的。概念解决不了这个问题。它们只能让“允许哪些形状”更清晰。

在设计泛型代码时，强迫自己把所有模型上都固定不变的运维成本，以及随模型变化的成本，都陈述出来。算法是否要求连续存储？它是否会物化中间 buffer？某个策略类型是否会被内联进每个翻译单元？某个概念是否同时接受抛异常和不抛异常的操作，从而把失败处理涂抹到整个接口上？

这些是设计问题，不是优化琐事。一个模板如果看起来很抽象，却只在某一类类型上性能良好，那么它往往就是一个不稳定的抽象。要么收窄概念，要么为成本模型有实质差异的情况提供不同的重载。

这一点对于跨大型代码库使用的头文件尤为重要。每增加一次实例化，都会增加编译成本，并且可能增加代码体积。如果某个组件跨越共享库或插件边界，那么普通虚接口或类型擦除 callable 往往是更好的权衡，即使它放弃了一些内联。稳定边界通常比理论上的“零开销纯度”更有价值。

## 普通重载何时优于概念

一直有一种诱惑：把使用概念当成设计“现代化”的证明。要抵制它。如果你只有三种已知输入类型，也没有证据表明这个集合会增长，那么重载往往更清晰。如果你需要运行时多态边界，那就直接用。如果变化只在测试里重要，那么函数对象或一个可 mock 的小接口，往往比一个泛型子系统更容易维护。

当且仅当下面这些条件都成立时，概念才最有力量：

- 算法确实适用于一族类型；
- 所需操作能够被狭窄地陈述出来；
- 调用方能从编译期拒绝中受益；
- 实现的成本模型依然清晰可读。

只要这些条件失效，模板就会很快变成负担。

## 失败模式与边界条件

泛型代码往往会以一些重复出现的方式失败。

第一种是无约束渗漏：一个泛型函数接受“任何像 range 的东西”，另一个期待“任何可写的东西”，很快整个代码库就会在那些原本并非为彼此设计的组件之间形成偶然兼容。概念应该收窄这些接缝，而不是把它们放宽。

第二种是约束重复。作者在多个辅助函数之间复制几乎一样的 `requires` 子句，直到接口根本无法演化。对重复出现的要求，应优先使用具名概念。具名概念是文档，而不只是语法压缩。

第三种是语义漂移。一个原本为清晰角色设计的概念，会因为又一个调用方需要又一个操作，而逐渐累积起无关要求。一旦出现这种情况，就拆分概念，或者拆分算法。不要让一个抽象沦为收纳各种“差不多相关”用例的垃圾场。

第四种是诊断表演：概念栈看起来很讲究，编译器输出却依然不可读。如果使用者无法判断自己的类型为什么没能建模这个概念，那就应当简化。好的泛型设计包括人们可以采取行动的失败信息。

## 验证与评审

对泛型代码的验证，不只是“实例化一次”。

- 当某个概念足够核心，值得有稳定示例时，为有代表性的正向和反向模型添加 `static_assert` 检查。
- 用少量在实质上不同的模型类型测试算法，而不是排一长串只有表面差异的变体。
- 评审概念是否命名了系统中的真实角色，而不只是若干操作的打包。
- 如果泛型组件位于高频头文件路径上，测量编译时间和代码体积。
- 确认错误行为、分配行为和所有权假设在受约束的边界上是可见的。

只有当被拒绝的程序，按照团队能够解释清楚的设计标准，确实是错误程序时，编译期拒绝才有帮助。否则，你只是围绕一个不清晰的接口造了一道复杂的门禁。

## 要点

- 只有当问题中的变化真实且持久时，才编写泛型代码。
- 用概念约束公共边界，这样实现可以保持普通且可读。
- 用语义角色给概念命名，而不是用偶然的语法特征命名。
- 优先使用狭窄的定制点，而不是开放式特化方案。
- 如果编译期多态让成本、诊断信息或边界变得更差，就改用重载或运行时抽象。
