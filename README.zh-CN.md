# Modern C++ in Practice

[English](README.md) | [中文](README.zh-CN.md)

我做这个项目，原因其实很简单：我想系统地学一遍现代 C++，结果发现市面上几乎没有真正靠谱、而且明确站在 C++23 语境里的课程或书。很多材料不是还停在 C++11/14 的思路里，就是把重点放在语法讲解、特性罗列和零散 demo 上。你看完之后也许知道了语言里“有什么”，但还是不知道工程里“该怎么写”。

所以我决定自己写一本。准确地说，是一边学，一边写，一边把内容往工程实战的方向打磨。Claude Opus 和 GPT-5.4 参与了不少起草和整理工作，但它们不是作者替身，更像高强度草稿机和讨论对象。最后定方向、改结构、做取舍、校正技术内容的人，还是我自己。

这个仓库就是这本书的写作现场。

## 我想写一本什么样的 C++ 书

这本书不是按语法点往下铺，也不是把标准库组件挨个点名。它关心的是另一件事：在真实项目里，现代 C++ 到底该怎么用，哪些写法值得坚持，哪些地方最容易失控，出了问题该怎么定位，做设计时又该怎么取舍。

书里会反复碰到这些主题：所有权和生命周期、RAII、值语义、不变量、错误边界、接口设计、并发、性能、测试、诊断、可观测性。不是为了把名词列全，而是因为这些东西会一起决定一段 C++ 代码最后是能长期维护，还是迟早失控。

每一章都会尽量从一个真实的工程问题切入。先看常见写法为什么在压力下会出问题，再看现代 C++ 的写法到底解决了什么，又引入了什么成本。如果某种方案有明显副作用，书里会直接讲，不会假装它是银弹。

## 这本书不打算写什么

这不是一本入门教材。

这里不会写 IDE 安装、编译器配置、"hello world"，也不会按“变量、循环、函数、类”这种顺序重新上一遍程序设计基础课。如果你要的是从零开始学语法，这个仓库不适合。

它也不是一本老项目迁移手册。重点不是“如何把一套十几年前的 C++ 代码逐步改造成现代风格”，而是“如果你今天重新写，或者正在维护一个还来得及做对的新系统，应该怎么思考”。

默认基线是 C++23。C++26 只会在一个前提下出现：它确实会改变你今天的设计判断。

## 适合谁看

如果你已经写过几年软件，知道线上系统、接口边界、测试、部署、回归、性能问题这些东西到底意味着什么，但对 C++ 还不够熟，或者以前学过一版、现在想重新补一遍现代写法，那这本书大概率适合你。

我默认读者是“工程上成熟，但 C++ 不一定成熟”。这也是整本书的写法基线。

## 仓库里有什么

- `chapters/`：书稿正文，按 mdBook 的结构组织。
- `examples/`：配套示例代码。有些可以直接编译运行，有些会刻意保留为局部片段，用来聚焦某个问题。
- `style-guide.md`：这本书的编辑约定，包括语言风格、示例代码风格和版本边界。

## 这些章节是怎么写出来的

我一般按下面这个节奏推进：

1. 先把这一章到底要回答什么问题写清楚。
2. 定边界：哪些内容该讲，哪些不该讲，示例场景放在哪个工程语境里更合适。
3. 按 style guide 起草。
4. 回头检查技术判断是否站得住脚，和前后章节有没有重复或冲突。
5. 能跑的示例就编译、运行、验证；不能完整运行的，就明确标注为什么只写成片段。
6. 最后再收语言，把“知识点说明”改成“工程决策说明”。

书里的例子会尽量贴近真实系统，比如服务端、库、数据处理管道、并发密集组件、内存敏感模块之类。不会为了演示一个语法特性，专门造一个除了教学之外没有任何意义的小玩具程序。
